# Refresh トリガー 一枚絵プロンプト

Gemini Nano Banana に「いつ更新（refresh）をリクエストするか」を一枚のフローチャートとして描かせるためのプロンプト。
関数名・変数名は伏せ、ユーザー視点での状態とタイミングだけで読めるように構成している。

---

## Nano Banana 用プロンプト（コピペ用）

```
横長 16:9 の一枚絵で、日本語のフローチャートを描いてください。
タイトル：「ツイート更新（refresh）はいつ走るか」
スタイル：白背景、フラットデザイン、角丸の矩形ノード、太めの矢印、
配色は淡い水色・淡い黄色・淡い緑・淡い赤の4色を機能ごとに使い分け、
文字は黒の太字ゴシック。手書き風ではなくきれいなインフォグラフィック調。

【全体構造】
左から右へ大きく3レーンに分ける：
  レーン①「起動〜閲覧開始」（淡い水色）
  レーン②「閲覧中の手動/復帰トリガー」（淡い黄色）
  レーン③「閲覧中の自動トリガー（AUTOモードのみ）」（淡い緑）
画面下部に独立した枠で「更新パイプライン（共通処理）」（淡い赤）を配置し、
①②③ のすべての発火点から点線矢印で同じ枠に集約させる。

【レーン①：起動〜閲覧開始】
ノードを縦に並べて矢印でつなぐ：
  1. 「アプリ起動」（丸型・スタートノード）
  2. 「裏で先行スクレイプを予約」
     ※小さい注釈：「結果は受け取らず、ID だけ控える」
  3. 「モード選択画面を表示」
  4. 「ユーザーがモードを確定（タップ／ダブルタップ）」
  5. 「★ ここで更新リクエスト発火①」（赤い★アイコン付き）
     ※注釈：「②で控えた ID を引き継いで完了を待つ」
  6. 「閲覧開始」

【レーン②：閲覧中の手動/復帰トリガー】
上から3つのトリガーを並列に並べ、それぞれ「★ 更新リクエスト発火」で終わる：
  A. 「フィード最上端で、さらに上へ引っ張った」
       → 「★ 更新リクエスト発火②（プル・トゥ・リフレッシュ）」
  B. 「アプリを別画面に切り替えて戻ってきた（フォアグラウンド復帰）」
       → 「★ 更新リクエスト発火③」
  C. 「（参考）バックグラウンドに行った瞬間」
       → 「自動更新タイマー停止」（★なし・グレー表示）
       ※注釈：「裏では更新しない＝通信節約」

【レーン③：閲覧中の自動トリガー（AUTOモードのみ）】
ひし形の判定ノードを縦に連ねた「ぜんぶ Yes なら発火」の連続ゲートにする。
入口に「AUTOモードで1ページ進むたびに毎回チェック」と書く。

  判定1：いまAUTOモードか？           → No なら「何もしない」へ
  判定2：このサイクルでまだ自動発火していないか？ → No なら「何もしない」
  判定3：いま更新処理が走っていないか？ → No なら「何もしない」
  判定4：前回更新から十分ページが進んだか？（目安：約3分以上）
                                       → No なら「何もしない」
  判定5：未読の新着ページが残りわずかか？（目安：残り約90秒分以下）
                                       → No なら「何もしない」
  判定6：「新着がまだ多めにある」 OR 「前回更新から十分時間が経った（約6分）」
                                       → No なら「何もしない」
  全部 Yes：
       → 「★ 更新リクエスト発火④（プリチャージ）」
         ※注釈：「読み終わるタイミングと
                  次の取得完了をぴったり揃える」

「何もしない」ノードは1つにまとめ、点線で「閲覧継続」に戻す。

【下部・共通処理ボックス（淡い赤）】
タイトル：「更新リクエストが走ると共通でこれをやる」
横長の角丸枠の中に、左から右へ番号付きで並べる：
  ① 「更新中フラグを立てる」
  ② 「スクレイプを依頼（既に予約済みならその ID を流用）」
  ③ 「完了するまで数秒おきに状況確認（最大10分）」
  ④ 「最新ツイートを取得」
  ⑤ 「既存と差分を取って、新着を先頭に挿入」
  ⑥ 「現在位置を先頭に戻す／自動発火フラグをリセット」

レーン①の★、②のA・Bの★、③の★、すべてから点線矢印で
この共通処理ボックスに合流させる。

【凡例（右下に小さく）】
  ●青：起動フェーズ
  ●黄：手動・復帰トリガー
  ●緑：自動トリガー（AUTOのみ）
  ●赤：共通の更新処理
  ★：更新リクエストが実際に飛ぶ瞬間

【描画上の注意】
- 矢印は重ならないように整理し、合流は1点にまとめる。
- 文字は全部日本語。英単語・関数名・変数名は一切出さない。
- 「★ 更新リクエスト発火」は4箇所すべて同じ赤い★アイコンで強調。
- ノード内の文章は最大2行に収め、長い注釈は小さめのフォントで脇に添える。
- 全体を一目で「どの瞬間に更新が走るか」が分かるレイアウトにすること。
```

## 補足：4つの「更新が走る瞬間」だけを抜き出すと

| # | きっかけ | ユーザーから見えるイベント |
|---|---|---|
| ① | モード選択でモードを確定したとき | 起動直後、ローディング → 一覧表示 |
| ② | フィード最上端でさらに上に引っ張ったとき | 手動のプル・トゥ・リフレッシュ |
| ③ | 別アプリから戻ってきたとき | フォアグラウンド復帰 |
| ④ | AUTO閲覧中、読み終わりが近づいたとき | 自動プリチャージ（無音で先回り） |

※ アプリ起動直後にも「裏でスクレイプ予約」は走るが、結果のマージは ① のタイミングで行われるため、ユーザーから見える「更新」は上記4回。

## 補足：自動トリガー④の発火条件（プロンプト内 判定1〜6 の元データ）

| 判定 | 具体値（プロンプトでは伏せている） |
|---|---|
| 4 | 前回更新から 18 ページ以上経過（≒約3分） |
| 5 | 未読の新着ページ残量が 9 ページ以下（≒約90秒） |
| 6 | 新着が 9 ページ超 **または** 前回更新から 36 ページ以上経過（≒約6分） |
